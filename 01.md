# CORD-01: Private Streams

Private streams can be created on Nostr between multiple parties by sharing a private key between all participants.

All participants sign NIP-59 giftwrap events with the shared key and publish them to relays. Participants with the key can query the stream by its public key:

```json
{ "kinds": [1059], "authors": ["<stream pubkey>"] }
```

To join a private stream, people share invite links. The invite link results in the private key.

## Stream Event

A stream event is a kind 1059 event. It appears to be a regular giftwrap, blending in with other giftwrap traffic, but it's actually not a regular giftwrap.

```json
{
  "id": "<id>",
  "kind": 1059,
  "pubkey": "<stream pubkey>",
  "content": nip44_encrypt(conv_key, {
    "id": "<id>",
    "kind": 20013,
    "pubkey": "<real author's pubkey>",
    "content": nip44_encrypt(conv_key, {
      "id": "<id>",
      "kind": 9,
      "pubkey": "<real author's pubkey>",
      "content": "Hey chat!",
      "tags": [],
      "created_at": 1686840217,
    }),
    "tags": [],
    "created_at": 1686840217,
    "sig": "<real author's signature>"
  }),
  "tags": [
    ["p", "<ephemeral pubkey>"]
  ],
  "created_at": 1686840217,
  "sig": "<stream signature>"
}
```

- Unlike NIP-59, which uses ephemeral authors and fixed "p" tags, streams use fixed authors and ephemeral "p" tags. It's reversed.
- Unlike NIP-59, the wrap is NIP-44-encrypted under the **stream conversation key** (self-ECDH of the shared stream key with the stream pubkey), never the "p"-tagged key.
- Unlike NIP-59, the seal event is kind 20013 (encrypted seal) or kind 20014 (plaintext seal, below) instead of kind 13 to avoid storage by relays.
- Unlike NIP-59, `created_at` should not be altered or tweaked.

> **Seal content, encrypted or plaintext.** Encrypting the rumor inside the seal (as the example does) makes it a non-standalone artifact, so no relay, honest or malicious, can retain and display it as a public event. But a signature over encrypted content is bound to the key that encrypted it, and breaks if that content is re-encrypted under a different key. So a stream that must carry a *signed* event across a key rotation leaves its rumor plaintext, and the signature survives re-wrapping; one that wants the leak-resistance keeps it encrypted. Which form applies is a fixed property of the protocol layered over the stream, per context, never a per-message choice, and the seal's kind declares it: an **encrypted seal is kind 20013**, a **plaintext seal is kind 20014**, so a reader never sniffs the content to learn its form. Concretely, the plaintext form is the same wrap with the seal's `content` holding the rumor verbatim, where the example above shows a `nip44_encrypt(...)` string. Verbatim means byte-verbatim: the seal's `content` is the rumor's serialized JSON *string*, and a re-wrap MUST carry those exact bytes forward, never re-serialize them, or two clients would compute different hashes from the same signed rumor.

## Binding

The wrap is signed by the shared stream key, so any keyholder can re-publish a decrypted seal at another stream address, the author's signature proves *who* wrote a rumor, not the *context* it was for. A protocol that splits a stream into sub-contexts (rooms, epochs) MAY bind them: commit the context as tags inside the signed rumor, and check them against the coordinate whose key opened the wrap. Without it, a keyholder can splice one author's rumor into a context they never chose.

## Deletions

To prevent participants from deleting each-other's giftwraps, relays should prevent giftwrap deletions by author. This is semantically correct for NIP-59, which already requires that relays *allow* deletion by the "p" tagged user. Giftwraps are owned by the recipient, not the sender. Therefore, clients should use at least one relay that rejects giftwrap deletions.

Users can delete their content in streams by sending giftwrapped kind 5 deletion events to the stream. Users can also delete their own giftwraps by "p" tag (on NIP-59 supporting relays) if the client saves the ephemeral key.

## Removing Participants

To remove a participant, a new stream must be created. This can happen automatically, with clients upgrading an existing chatroom to a new stream. Events would be combined from both the old and new streams into a single chatroom.
