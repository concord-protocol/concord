# CORD-04: Audio/Video

`draft` `optional`

Voice calls can be added to any channel without a host or a roster. There is no server that can check membership before handing out media tokens, so clients prove possession of the channel key instead: anyone holding the key can derive a signing key, fetch a short-lived token from a broker, and connect to a WebRTC SFU (e.g. [LiveKit](https://livekit.io)). Media is end-to-end encrypted, so the broker and SFU only ever forward ciphertext.

## Voice Keys

A voice channel is any channel (CORD-02) with voice enabled. Two sub-keys are derived from the channel's key:

```js
voice_signer_seed = hkdf(chat_key, "concord/voice-signer")
voice_media_key   = hkdf(chat_key, "concord/voice-media")
```

- `voice_signer_seed` becomes a secp256k1 secret key `voice_signer_sk`. If the seed is not a valid scalar, append an incrementing counter byte and re-derive. Its x-only pubkey is the SFU room name (64-char lowercase hex).
- `voice_media_key` is the 32-byte symmetric key for media encryption.

Both keys change whenever `chat_key` changes: on each epoch (CORD-02) and when a participant is removed via a new stream (CORD-01). When that happens, the room name and media key roll over and clients rejoin the new room. Until then, a removed participant can still derive both keys and join calls.

## The Broker

A broker is a simple service that mints SFU tokens. It holds no community secrets and keeps no state other than a transient anti-replay set.

```
GET /.well-known/concord/av              → 204 No Content   (capability probe)
GET /.well-known/concord/av/<voice_room> → { "token": "<jwt>", "url": "<sfu ws url>" }
```

To request a token, the client signs a NIP-98-style event (kind 27235) with `voice_signer_sk`, so `event.pubkey` equals the room name, and sends it base64-encoded in the header `Authorization: Concord <base64(event)>`:

```json
{
  "kind": 27235,
  "pubkey": "<voice_room>",
  "content": "",
  "tags": [
    ["u", "https://broker.example/.well-known/concord/av/<voice_room>"],
    ["method", "GET"]
  ],
  "created_at": 1686840217,
  "sig": "<signature by voice_signer_sk>"
}
```

The broker accepts if:

- The signature is valid.
- `event.pubkey` equals `<voice_room>` from the path.
- The `u` and `method` tags match this exact request.
- `created_at` is within ±60s.
- The event id has not been seen before (ids are retained for at least twice the freshness window).

It then mints a short-lived token scoped to joining that one room. Only a holder of the channel's key can derive `voice_signer_sk`, so the broker needs no lookup and no community knowledge. The broker assigns each participant a random SFU identity and never learns who is joining. The broker and SFU still see IPs and connection timing; members who need those hidden should use an anonymizing transport.

## Media Encryption

Media must be end-to-end encrypted: each member feeds `voice_media_key` to the SFU client's key provider before publishing. Every member derives the identical key, so there is no in-band exchange.

The media layer has no sender authentication among key-holders. Attribution rests on the SFU identity being broker-assigned and single-use, which is sound unless a member colludes with the broker or SFU.

## Presence

Who is in voice is announced over the channel itself as ordinary stream events (CORD-01), so relays and brokers stay blind. The innermost event:

```json
{
  "kind": 3306,
  "pubkey": "<member real pubkey>",
  "content": "",
  "tags": [
    ["channel", "<channel pubkey>"],
    ["voice", "1"],
    ["status", "joined"],
    ["identity", "<SFU identity>"],
    ["broker", "https://broker.example"]
  ],
  "created_at": 1686840217
}
```

- On join and every ~30s heartbeat, publish a `joined` carrying the assigned SFU `identity` and the `broker` origin. On leave, best-effort publish a `left` (identity and broker omitted).
- Per author, the latest presence wins. A `joined` older than ~90s is stale and counts as absent.
- Clients render an SFU participant as a member only if a fresh signed presence claims that exact identity. Anything else is shown as unverified.

## Rendezvous

The `broker` tag on live presence lets members converge on one room with no configuration:

- If anyone is present, join their broker.
- On a tie, pick the origin with the smallest `sha256(voice_room + origin)`.
- If the room is empty, use your own preferred broker.
- Probe the capability endpoint first and fall back if unreachable.

The hint is untrusted input from a fellow member, not an authorization. A malicious member can steer the call to a broker/SFU of their choosing, which then sees IPs and timing, but E2EE means it can never decode the media.

## Video and Screenshare

Video and screenshare are the same channel: a member additionally publishes camera or screen tracks to the room, under the same media key and the same presence. No additional keys or events are defined.
