# CORD-03: Channels

A Channel is one message plane inside a Community, *defined* in the Control Plane but carrying its own chat on an independent Chat Plane (CORD-02), a Community is simply a set of these unified under a common Control Plane.

Channels come in two kinds, differing only in *who* may read them:

**Public**: readable by every member, its key is *derived* from the `community_root`, so nothing is ever delivered and it rotates for free whenever the base does.

**Private**: readable only by granted role-holders, its key is an *independent* random secret, delivered on grant and rekeyed on removal (CORD-06).

----

## 1. Keying

Every Channel is addressed and encrypted the same way, through the one `group_key` derivation (CORD-02), only its *secret* and *epoch* differ by kind:
```
Public  channel_pk = group_key("concord/channel", community_root, channel_id, root_epoch).pk
Private channel_pk = group_key("concord/channel", channel_key,    channel_id, channel_epoch).pk
```
Because `channel_id` is folded into the derivation, every Channel gets its own distinct address regardless of which secret feeds it, so a Public Channel is really just "a Channel whose key derives from the `community_root`" rather than a separate mechanism.

A Public Channel needs no access key generation nor distribution, it's inherently readable by every member by definition, so deriving from the `community_root` means it needs no delivery, adds nothing to an invite, and a single Refounding (CORD-06) severs it along with the base, no per-channel rekey. A Private Channel needs real independence (a leaked channel key exposes only that one Channel, and it can be rotated independently), so it carries its own key and its own epoch.

## 2. Metadata

A Channel is *defined* by a **ChannelMetadata** entity in the Control Plane (CORD-04), holding its `channel_id`, name, `private` flag, and optional `voice` flag (marking it callable, CORD-07, absent means false), and edits fold by the same versioned editions as any control state, so a rename or a visibility change is an authorized, convergent edit rather than a new Channel. The edition's content (the `channel_id` is already the edition's `eid`, CORD-04), plus the optional client-extensible `custom` object (CORD-02 §6):
```jsonc
{ "name": "general", "private": false }                  // name ≤ 64 bytes, the protocol-wide cap (CORD-04)
{ "name": "lounge",  "private": false, "voice": true }   // a callable Channel (CORD-07)
```
A Channel is *created* by minting a random 32-byte `channel_id` and publishing this entity's first edition (`MANAGE_CHANNELS`, the owner at genesis), a Private one mints and delivers its independent key alongside. A Channel is *deleted* by an edition setting `"deleted": true`, and deletion is terminal: the id is never reused, clients drop the Channel from display and may discard its keys, though its history stays decryptable to anyone who already held them, deletion, like privatising, can't unshare the past.

Converting a Public Channel to a Private Channel is a key rotation: the Channel mints its own independent key at the **next channel_epoch** (monotonic, never resetting, so the first privatisation is epoch 1 and each later one climbs), delivered to the desired role-holders, and its metadata flag flips, both bind as a single authorized action (`MANAGE_CHANNELS`). The `channel_id` never changes across any conversion, it's the Channel's permanent identity (its ChannelMetadata `eid`, CORD-04), only the key and epoch turn over. Because the epoch only ever climbs, privatise → publish → privatise is safe with no extra bookkeeping: each private generation lives at a distinct epoch, so a stale key (always a lower epoch) can never share a coordinate with the current one, and a joiner is simply handed the current `(key, epoch)` in its invite (CORD-05). Privatising protects the *future* only: the pre-conversion history was written under `community_root`-derived keys every member holds, so it stays readable to all members (it was public when written), no conversion retroactively hides what was already shared.

Converting from Private to Public reverses it, the Channel simply begins deriving from the `community_root` going forward, and a member joining after the switch reads only the now-public history, never the prior private messages (since they never held that key).

## 3. Messages

A Channel's Chat Plane carries the ordinary append events (messages, reactions, edits, deletes), each wrapped exactly as a Stream event (CORD-01), signed by the Channel's `group_key` and sealed to the author's real identity within:
```json
{ "kinds": [1059], "authors": ["<channel_pk per held epoch>"] }
```
Clients load a Channel newest-first and paginate backwards, querying every epoch pubkey they hold (a Public Channel across the base epochs, a Private one across its own), so history spanning a rekey stays continuous.

Concord Channels take up CORD-01's binding as a requirement: each message MUST commit `["channel", channel_id]` and `["epoch", n]` inside the author-signed rumor, and a receiver MUST check both strict-equal against the Channel and epoch whose key decrypted the wrap, dropping a mismatch. So no member can re-wrap another's message into a different Channel or replay it across an epoch.
